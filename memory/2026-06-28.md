## QQ机器人故障修复（09:00-10:50）

### 问题1：DeepSeek API 连接失败
- **症状**：QQ收到消息后，OpenClaw调用DeepSeek时一直报 `LLM request failed: network connection error. rawError=Connection error.`，重试4次后超时
- **根因**：系统环境变量 `HTTPS_PROXY=socks5://127.0.0.1:9567` 被 Node.js 进程继承。OpenClaw 内部使用 Node.js undici/fetch 无法正确处理 socks5 代理的 HTTPS 请求，导致 "Connection error" 
- **影响**：不仅 DeepSeek 连不上，连 QQ 官方 API (`bots.qq.com`, `api.sgroup.qq.com`) 也 `fetch failed`
- **修复**：在 `Windows-Start.bat` 启动脚本中加了 `set HTTP_PROXY=` 和 `set HTTPS_PROXY=` 清除代理环境变量

### 问题2：QQ机器人配置丢失
- **根因**：我 kill 旧 gateway 进程时，config server 重新读取 openclaw.json 并覆盖了文件，导致 channels.qqbot 和 plugins.entries.qqbot 配置丢失
- 新版 OpenClaw 2026.3.24 的 plugins.entries schema 不认 `source` 字段（旧版要 `source`，新版只要 `enabled: true`）
- **修复**：重新写入完整 openclaw.json，去掉了 `source` 字段，加了完整的 qqbot channels 配置
- **最终验证**：QQ 机器人正常上线，网页回复 + 语音回复均正常

### 经验教训
- `config.patch` 在 Windows 上可能因为 SIGUSR1 不生效，下次直接改文件然后杀进程重启
- 新版 OpenClaw 不认 `plugins.entries.qqbot.source` 字段
- 系统 socks5 代理会影响 Node.js 的 HTTPS 请求

### 三次注入尝试与失败总结（19:00-20:20）

**试次1（v1, 19:02-19:07）**：`--custom-gadget-name libtersafe_hook.so`
- 注入成功，APK签名成功（1.44GB）
- 问题：so文件名变成 `libtersafe_hook.so.so`，loadLibrary("tersafe_hook.so")虽然能加载，但黑屏
- 根因：名字太接近ACE的libtersafe.so（5.3MB），可能被误杀

**试次2（v2, 19:50-20:07）**：默认gadget名 + 内嵌 `assets/frida-gadget.config`
- 手动反编译→加config→重编译→签名
- 问题：assets里的config文件gadget不自动读，依然黑屏

**试次3（v3, 20:11-20:15）**：默认gadget名（无custom name）
- so文件：`libfrida-gadget-17.15.3-android-arm64.so`（24MB）
- 注入代码：`loadLibrary("frida-gadget-17.15.3-android-arm64")` → 正确匹配文件名
- 结果：**依然黑屏**

**根本原因判断**：腾讯ACE反作弊在校验APK签名完整性。一旦发现APK被重新签名（debug key vs 原厂签名），ACE主动挂起游戏防止作弊。无论gadget怎么改名，只要APK签了debug key就会触发。

**结论**：frida-gadget免root方案对三角洲行动（腾讯ACE保护）无效。
